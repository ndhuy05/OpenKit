You are creating a README.md file for the current project. Your goal is to make it appealing, informative, and easy to read.

## Task

1. Take a deep breath, and review the entire project and workspace, then create a comprehensive and well-structured README.md file for the project.
2. Take inspiration from these readme files for the structure, tone and content:
   - https://raw.githubusercontent.com/Azure-Samples/serverless-chat-langchainjs/refs/heads/main/README.md
   - https://raw.githubusercontent.com/Azure-Samples/serverless-recipes-javascript/refs/heads/main/README.md
   - https://raw.githubusercontent.com/sinedied/run-on-output/refs/heads/main/README.md
   - https://raw.githubusercontent.com/sinedied/smoke/refs/heads/main/README.md
3. Do not overuse emojis, and keep the readme concise and to the point.
4. Do not include sections like "LICENSE", "CONTRIBUTING", "CHANGELOG", etc. There are dedicated files for those sections.
5. Use GFM (GitHub Flavored Markdown) for formatting, and GitHub admonition syntax (https://github.com/orgs/community/discussions/16925) where appropriate.
6. If you find a logo or icon for the project, use it in the readme's header.

## What You'll Do

**Explore the project**
- Read package.json for project name, description, scripts, dependencies
- Check for existing docs (docs/), config files, and main entry points
- Identify the tech stack and key features
- Look for any visual assets (logos, icons)

**Structure the README**
- Title + one-liner description
- Badges (if appropriate)
- Quick start / installation
- Features (key capabilities)
- Architecture / how it works
- Configuration (if applicable)
- Contributing guidelines reference
- Links to other docs

**Write with best practices**
- Start with what the project does (not what it is)
- Show, don't just tell — include code snippets
- Group related info, use clear headings
- Keep it scannable with lists and tables
